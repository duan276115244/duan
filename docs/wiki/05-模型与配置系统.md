### 智能选择算法

## ModelRouter

### `src/core/model-router.ts`（748 行）

与 ModelLibrary 的差异：ModelRouter 是**面向 Agent Loop 工具**的"评分/成本估算"层，不做实际 API 调用。

**核心选择算法** `selectModel(task, complexity, requirements?)` (L293-380)：
1. `analyzeTaskCapabilities(task)` → 所需能力标签
2. `filterByRequirements` 过滤候选
3. `adjustWeightsByComplexity`：
   - simple → 质量 0.2 / 速度 0.5 / 成本 0.3
   - medium → 质量 0.4 / 速度 0.3 / 成本 0.3
   - complex → 质量 0.6 / 速度 0.2 / 成本 0.2
4. `compositeScore = quality*Qw + speed*Sw + cost*Cw`
5. 估算延迟和成本
6. 记录 `selectionHistory`（保留 100 条）

**3 个 LLM 工具**：`model_select` / `model_list` / `model_cost`

## 统一配置

### `src/core/unified-config.ts`（786 行）

**核心定位**：三端（CLI / Web / Desktop）配置互通的唯一配置源，`~/.duan/config.json`（v2.0 格式），AES-256-GCM 加密 API Key，fs.watch 实时同步。

**配置结构**：
```typescript
{
  version: '2.0',
  profiles: Record<string, UnifiedProfile>,  // provider/apiKey(加密)/model/baseUrl/label
  activeProfile: string,
  preferences: { theme, language, autoApprove },
  sync: { lastModified, deviceId },
  mobileChannels?, channels?, webPort?, workspace?
}
```

### 加密机制

`ConfigEncryptor` (L110-161)：
- 算法：**AES-256-GCM**
- 主密钥派生 `deriveMasterKey()` (L100)：`scryptSync('duan-unified-config:${hostname}:${username}', 'duan-aes-256-salt', 32)` — 同一台机器三端密钥一致
- 加密格式：`enc:<iv_hex>:<authTag_hex>:<ciphertext_hex>`
- 降级：加密失败用 `enc:base64:<base64>`，绝不裸露存储
- 解密失败返回空字符串（而非密文）

### 文件监听

`startWatch()` (L451-478)：
- `fs.watch` + 200ms 防抖
- `selfWrite` 标志跳过自身写入触发的事件
- 错误时 5 秒后自动重试
- 冲突解决：比较 `sync.lastModified`，最新的胜出

### 供应商热切换机制

```
用户切换 profile (Web/Desktop/CLI)
    ↓
UnifiedConfigManager.setActiveProfile()
    ↓ persist() 写入 config.json
    ↓ broadcastChange('config.profile.activated')
    ↓ broadcastChange('config.external.changed')  ← P0 修复
    ↓
EventBus → ModelLibrary.reloadFromUnifiedConfig()
    ↓ loadFromUnifiedConfig() 重载 apiKey
    ↓ clients.clear()  清理旧客户端缓存
    ↓
下次 callModel 时 initClient() 用新 apiKey 重建
```

`migrateFromV1()` (L569-631)：v1.x profiles 为数组 → v2.0 为对象映射

## 数据安全

### 原子写 `src/core/atomic-write.ts`

```typescript
atomicWriteJson(filePath, data): Promise<void>      // 异步原子写
atomicWriteJsonSync(filePath, data): void            // 同步原子写
```

**流程**：
1. 写入 temp 文件：`${filePath}.${process.pid}.tmp`（同目录，确保同文件系统）
2. `rename(temp, final)`（POSIX 原子；Windows 上 rename 覆盖目标）
3. 失败时清理 temp 文件

### 损坏防护 `src/core/corruption-guard.ts`

**主入口** `checkOnStartup(options?)` (L356)：启动时扫描 `~/.duan/` 和 `.awareness/` 下的 JSON 文件

**非破坏性修复** `repairCorruptJson(filePath)` (L258)：
1. **预检**：完好文件直接 no-op（防止误调破坏数据）
2. **备份**：`.corrupt.${Date.now()}.bak`
3. **选默认值** `pickSafeDefault()`：
   - 父目录 + basename 联合感知覆盖（`PARENT_DIR_OVERRIDES`）
   - `ARRAY_DEFAULTS_BY_BASENAME`：30+ 已知顶层数组文件
   - `ARRAY_DEFAULTS_BY_PREFIX`：`feedback-*` / `reward-*` 动态命名
   - 回退 `{}`
4. **原子写**重建
5. **验证**
6. **清理旧备份** `pruneBackups()`：保留最近 5 个

### 一致性防护 `src/core/consistency-guard.ts`

**6 种约束类型**：`factual | logical | naming | style | api_contract | data_schema`

**5 个内置约束**：
- `naming_consistency`：变量/函数名风格混用检测
- `api_contract`：函数签名变化检测
- `factual_consistency`：事实矛盾检测
- `style_consistency`：代码风格一致性
- `data_schema`：interface/type 字段变化检测
