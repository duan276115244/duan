# 03 - 工具系统

本章覆盖工具的定义、注册、智能选择、执行管道、熔断器和沙箱隔离机制。

## 概览

项目采用**多层级、可扩展的工具架构**，核心由 4 套互补系统组成：

| 系统 | 文件 | 职责 |
|------|------|------|
| ScalableToolRegistry | `src/core/scalable-tool-registry.ts` | 高性能运行时注册表（主执行路径） |
| UnifiedToolFramework | `src/core/unified-tool-framework.ts` | 完整生命周期管理（统计/审批/分类） |
| SmartToolSelector | `src/core/smart-tool-selector.ts` | 意图驱动的工具选择器 |
| ToolExecutionPipeline + FaultTolerantExecutor + CircuitBreaker | `src/core/` | 执行管道与韧性保护 |

工具总量约 **140+ 个内置工具**（34 个 built-in 文件）+ **51+ 个核心模块**通过 `getToolDefinitions()` 契约注册。

## 工具定义契约

### `UnifiedToolDef`（`src/core/unified-tool-def.ts` L25-L62）

```typescript
export interface UnifiedToolDef {
  // 基础字段（必需）
  name: string;
  description: string;
  parameters: Record<string, { type: string; description: string; required?: boolean }>;
  execute: (args: any, context?: ToolExecutionContext) => Promise<string>;
  readOnly?: boolean;
  // 分类字段（可选）
  id?: string; category?: ToolCategory; priority?: number; tags?: string[];
  // 安全字段（可选）
  riskLevel?: ToolRiskLevel; executionPolicy?: ExecutionPolicy; sandbox?: SandboxMode;
  // 性能字段（可选）
  minModelTier?: ModelTier; enabled?: boolean;
  // 延迟初始化（可选）
  init?: () => Promise<void>;
}
```

### `getToolDefinitions()` 契约

每个核心模块暴露此方法，返回 `ToolDef[]`。`bootstrap.ts` 的 `standardToolModules` 数组（L1982-L2044）统一遍历 51 个模块调用此方法。

### 两条注册路径

1. **内置工具路径**：`src/tools/built-in/*.ts` → `allBuiltInTools` 数组 → `createToolRegistry()` → `ScalableToolRegistry.register()`
2. **模块工具路径**：`CoreModules` 的 `getToolDefinitions()` → `createAgentLoop()` 的 `standardToolModules` → `loop.registerTools()`

### 风险等级

`TOOL_RISK_MAP`（`unified-tool-def.ts` L70-L92）是全项目唯一风险等级映射，禁止本地定义重复 riskMap。

分类推断 `inferCategory()`（L140-L151）：精确名称匹配 > 前缀匹配 > `'other'`。

## 智能工具选择器

### `src/core/smart-tool-selector.ts`

**8 种意图类型**（L26）：`code | browse | desktop | search | file | chat | self_modify | mixed`

### 选择流程

**Step 1: 意图推断** `inferIntentWithConfidence()` (L833-L881)
- 使用 `tokenize()` 分词
- 对 8 种意图计算加权评分：token 精确匹配 +2.0×weight，substring +1.0×weight
- 最高分意图与次高分差距 < 15% → 判定为 `mixed`

**Step 2: 工具选择** `selectTools()` (L551-L684)
1. 类别过滤：根据意图白名单过滤
2. 失败排除：排除连续失败 ≥3 次的工具
3. 必备工具注入：`essentialsByIntent[intent]`
4. 多维排序：计划提示 + RL 权重 + 用户画像 + 意图优先级 + 风险 + 相关性 + 近期使用
5. 冲突移除
6. 截断（默认 maxTools=15）

**Step 3: LLM Fallback**（Phase D2，L900-L936）
- confidence < 0.5 或 mixed + 提供 `llmCaller` → 调用 LLM 从 30 个工具 schema 中选 3 个
- 5min 缓存
- LLM 失败 → 兼容回退到 `selectTools('mixed', ...)`

## 执行管道

### ToolExecutionPipeline（6 阶段确定性管线）

`src/core/tool-execution-pipeline.ts`：

1. **Schema 校验**（L260-L329）：必填字段、类型、正则模式、枚举值
2. **路径穿越防护**（L335-L423）：检查 `..` 穿越和工作区边界
3. **危险命令拦截**（L429-L490）：11 个正则模式（rm -rf, mkfs, dd, curl|sh, fork bomb 等）
4. **审批门控**：high/critical 风险需审批，low/medium 自动通过
5. **沙箱执行**：根据风险等级选择隔离级别（process/vm/none）
6. **执行后审计**（L496-L546）：记录所有执行，敏感字段脱敏

### ToolOrchestrationEngine（多工具编排）

`src/core/tool-orchestration-engine.ts`：

**5 种步骤类型**（L27）：`tool | parallel | conditional | transform | delay`

**3 个预定义模板**：
- `read-analyze-modify`：读取→分析→修改→验证
- `search-read-summarize`：搜索→并行读取→摘要
- `test-fix-retest`：测试→条件判断→修复→重测

**4 个 LLM 工具**：`orchestrate_pipeline`、`orchestrate_template`、`list_orchestration_templates`、`orchestration_stats`

### FaultTolerantExecutor（容错执行）

`src/core/fault-tolerant-executor.ts`：

- **并行执行**：自动分析依赖，并行执行独立步骤
- **自动回滚**：关键步骤失败时逆序回滚
- **错误恢复**：指数退避重试（1s, 2s, 4s, 8s + 500ms 抖动）
- **异常检测**：错误模式、安全模式、意外模式
- **持久化**：历史保存至 `.duan/execution/history.json`

## 熔断器

### 三层熔断保护

**第一层：ScalableToolRegistry 内置熔断**
- 连续 3 次失败 → 禁用 5 分钟（`CIRCUIT_FAILURE_THRESHOLD=3`, `CIRCUIT_COOLDOWN_MS=5*60*1000`）
- 成功率指数移动平均：`successRate = successRate * 0.9 + (0|1) * 0.1`

**第二层：独立 CircuitBreaker**（`src/core/circuit-breaker.ts`）

3 状态机：`closed → open → half_open`

- **SlidingWindow**（L65-L165）：60s 窗口，最少 5 次调用，head 指针 + 惰性压缩
- **双重熔断判定**（L352-L374）：
  - 绝对失败数 ≥ 5（`failureThreshold`）
  - 滑动窗口错误率 ≥ 50% 且调用数 ≥ 5
- **半开状态**：限制 1 个请求，连续 3 次成功 → 关闭
- **指数退避**：`min(baseDelay * 2^(attempt-1), maxDelay) + jitter`
- **Bulkhead**（L478-L527）：资源隔离，限制并发 + 队列（默认 5 并发，50 队列）
- **ResilienceChain**（L540-L596）：组合 CircuitBreaker + Bulkhead

**第三层：FaultTolerantExecutor 步骤级熔断**
- per-tool-name 熔断器
- 灰度开关 `USE_STEP_CIRCUIT_BREAKER=true`
- 熔断打开 → 跳过该步骤剩余重试

### 工具执行韧性包装

`executeToolResiliently()`（`circuit-breaker.ts` L637-L702）：
- 始终生效：超时保护
- 灰度开关 `USE_TOOL_CIRCUIT_BREAKER=true` 启用熔断
- 熔断打开 → 快速失败返回 `❌ 工具 X 已熔断`

## 沙箱执行

### `src/core/sandbox-executor.ts` — 4 级隔离

| 级别 | 说明 | 适用 |
|------|------|------|
| `none` | 无隔离 | 仅限安全工具 |
| `vm` | Node.js vm 模块隔离 | moderate 风险工具 |
| `process` | 子进程隔离，委托 CrossPlatformSandbox | high 风险工具 |
| `docker` | 容器隔离 | 最高风险 |

**跨平台沙箱**：
- Linux: bubblewrap / unshare
- macOS: seatbelt
- Windows: Job Object

**安全措施**：
- 命令黑名单（rm -rf /, fork bomb, curl | sh 等）
- 敏感环境变量过滤（API_KEY/SECRET/TOKEN 等 9 类前缀）
- 工具风险分级自动选择级别：safe→none, moderate→vm, high→process

## 工具执行完整链路

```
LLM 请求工具调用
  → EnhancedAgentLoop 分发
  → tools.ts executeTool
  → ScalableToolRegistry.executeTool()
    → getTool() [熔断检查 + 延迟初始化]
    → 依赖检查
    → 发射 TOOL_CALL_START 事件
    → 4 层超时解析
    → 沙箱路由
    → executeWithTimeout()
    → recordSuccess/Failure [更新统计/熔断状态]
    → 发射 TOOL_CALL_COMPLETE 事件
  → UnifiedToolFramework.recordExternalExecution() 同步统计
```

**高风险审批路径**：
```
ScalableToolRegistry.executeWithApproval()
  → checkApproval()
    → autoApproveSafe: safe/moderate 自动放行
    → 信任命令精确/前缀匹配
    → PermissionRule 规则匹配
    → 审批回调
  → executeTool()
```
